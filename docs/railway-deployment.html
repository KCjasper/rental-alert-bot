Phase 6 · Railway Deployment

Railway 部署操作文件

本文件用於第 6 階段：把已通過本機端到端驗收的租屋情報機器人部署到 Railway， 並確認秘密變數、SQLite 持久磁碟、健康檢查、重啟政策與支出上限都已設定。

部署前狀態

第 5 階段已通過：本機服務累積 8.81 小時、100 次有效檢查、0 次失敗、0 筆重複成功通知，20 筆圖片人工抽查皆正確。

第 6 階段完成前，Railway 端仍必須實際完成一次重新部署與一次強制重啟，並確認 SQLite 訂閱與去重資料沒有遺失。

Repo 內部署設定

檔案用途
railway.json指定 Railway 使用 Railpack、啟動本機完整服務、設定 /health、自動重啟策略。
scripts/run_local_service.py同一程序內啟動 Telegram long polling、591 排程監控與健康檢查 HTTP server。
scripts/check_phase6_readiness.py部署前檢查 repo 設定；部署後可檢查 Railway runtime variables。

Railway 專案設定

  1. 在 Railway 建立新 Project，從 GitHub repository KCjasper/rental-alert-bot 建立服務。
  2. 確認服務使用 repository 根目錄，部署設定會讀取 railway.json。
  3. 建立 Volume，Mount path 設為 /app/data。SQLite 檔案必須寫入這個資料夾才會在重新部署後保留。
  4. 在 Variables 設定下表所有值。Telegram Token 與個人 ID 不要貼進聊天或提交 Git。
  5. 在 Railway Billing / Usage 設定支出上限；此項需在 Railway UI 人工確認。

Railway Variables

變數正式值說明
TELEGRAM_BOT_TOKENBotFather Token建議在 Railway UI 中設為 sealed variable。
AUTHORIZED_TELEGRAM_USER_ID你的 Telegram User IDBot 只接受這個 ID 的操作。
DATABASE_PATH/app/data/rental_alert.db必須位於 Volume mount path 內。
POLL_INTERVAL_SECONDS300每 5 分鐘檢查一次。
POLL_JITTER_SECONDS30加入 0–30 秒隨機延遲。
REQUEST_TIMEOUT_SECONDS20HTTP 請求逾時。
INITIAL_NOTIFICATION_BATCH_SIZE10首次全量通知分批大小。
TELEGRAM_SEND_DELAY_SECONDS1.2Telegram 發送節流。
FAILURE_ALERT_THRESHOLD3連續失敗幾次後告警。
LOG_LEVELINFO避免 DEBUG 記錄過多資訊。
TZAsia/Taipei服務時區。
HEALTHCHECK_PATH/health需與 railway.json 相同。
PORTRailway 自動提供不需手動設定；程式會在此 port 啟動健康檢查。

不要在 Railway Variables 填入本機 ./data/rental_alert.db。這會讓 SQLite 寫在非持久路徑，重新部署後資料可能遺失。

預檢與部署後檢查

推送前先在本機確認 repo 設定完整：

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\check_phase6_readiness.py

部署後可在 Railway shell 或一次性指令中檢查 runtime variables：

uv run --frozen python scripts/check_phase6_readiness.py --runtime-env

健康檢查應回傳 HTTP 200 並包含 {"status": "ok"}：

curl https://你的-railway-domain/health

第 6 階段驗收

項目通過條件
首次部署服務啟動成功，Railway healthcheck 顯示健康。
Telegram 操作/start、/subscriptions、/test 可正常回覆。
資料持久化新增或保留訂閱後，重新部署與強制重啟都不遺失訂閱與已通知紀錄。
重啟政策服務異常退出後 Railway 會自動重啟。
支出上限Railway Billing / Usage 已設定可接受的上限。
日誌安全Deploy logs 不包含 Telegram Token 或完整秘密值。

第 6 階段不能只靠 repo 設定宣告完成；必須在 Railway 上完成一次 redeploy 與 restart 後驗證資料仍存在。
