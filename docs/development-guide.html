Developer Guide · Phase 1–4

開發環境操作指南

本文件說明如何建立隔離環境、執行品質檢查，以及驗證應用程式啟動設定。

階段 1–4 已通過；階段 5 開發中，長時間驗收工具已完成

環境需求

工具需求
Python3.12 至 3.14；專案預設使用 3.12。
uv負責建立虛擬環境、安裝鎖定依賴及執行命令。
Git版本管理與 GitHub 同步。

第一次安裝

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv sync --frozen

指令會建立專案專用的 .venv，不會修改系統 Python 套件。

執行完整檢查

powershell -ExecutionPolicy Bypass -File .\scripts\check.ps1

檢查順序為：鎖定依賴、Ruff 靜態檢查、pytest 自動測試、Python 編譯檢查。

驗證啟動設定

在專案根目錄建立未提交的 .env：

TELEGRAM_BOT_TOKEN=
AUTHORIZED_TELEGRAM_USER_ID=
DATABASE_PATH=./data/rental_alert.db
UV_CACHE_DIR=.uv-cache

把 BotFather 給你的 Token 填在 TELEGRAM_BOT_TOKEN 等號後面，把自己的 Telegram User ID 填在 AUTHORIZED_TELEGRAM_USER_ID 等號後面。

填好後直接執行：

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen rental-alert-bot --check
不要把真實 Token 寫入文件、Git 或聊天。正式值只放在未提交的 .env 或 Railway Variables；程式會自動讀取專案根目錄的 .env。

初始化資料庫

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\init_database.py

預設建立 ./data/rental_alert.db。指令可重複執行，只會套用尚未執行的 migration；目前 schema 版本為 v5。

備份與還原

uv run --frozen python .\scripts\backup_database.py `
  --destination .\backups\rental-alert-2026-06-15.db

備份使用 SQLite online backup API 並執行完整性檢查。若目標檔已存在，工具會停止， 不會覆蓋原備份；備份來源資料庫不會在備份前自動 migration。

還原時先停止服務，保留目前資料庫，再將設定指向新的還原檔：

$env:DATABASE_PATH = ".\data\rental-alert-restored.db"
uv run --frozen python .\scripts\init_database.py --path $env:DATABASE_PATH
請先把備份檔複製成新的還原檔名，再設定 DATABASE_PATH。 不直接覆蓋或刪除目前資料庫，確認服務可讀取後再人工處理舊檔。

手動測試 Telegram Bot

先由使用者透過 BotFather 建立私人 Bot，Token 不要貼進聊天；只放在本機 .env 或 Railway Variables。User ID 也只存設定，不提交 Git。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\run_telegram_bot.py
手動驗證項目通過條件
非授權帳號傳訊息收到拒絕訊息，資料庫無新增資料。
貼上 591 搜尋網址Bot 顯示目前房源數量，並等待回覆「確認」。
回覆「確認」分批收到現有房源，完成後訂閱變成啟用。
/cancel取消等待確認、等待輸入編號的新增或刪除動作。
/pause、/resume、/test、/deleteBot 先提示輸入訂閱編號，回覆數字後成功執行；錯誤案例都有清楚回覆。
刪除後再查看與測試/subscriptions 不再顯示已刪除訂閱，/test 不能再測試已刪除訂閱。

完成上述操作後，用本機資料庫檢查階段 4 證據：

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\validate_phase4_manual.py

顯示 PHASE4_MANUAL_VALIDATION_OK 才代表階段 4 實機驗證完成。 這個檢查只讀取指令類型、授權狀態與訂閱狀態，不保存 Telegram 訊息原文。

階段 4 已完成私人 Bot 實測並通過驗證腳本；後續若修改 Telegram 指令流程，需重新執行本段手動驗證。

執行一次新房源監控

階段 5 開始後，可先用一次性腳本驗證目前到期的 active 訂閱。它會讀取 .env、抓取 591、寫入 SQLite，並只通知尚未通知的新房源。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\run_monitor_once.py

這不是長時間常駐排程；常駐排程與本機完整服務已另有腳本，8 小時穩定性驗收仍屬階段 5 後續工作。

啟動常駐新房源監控

常駐監控會反覆檢查已到期的 active 訂閱；每個訂閱的下一次檢查時間由 POLL_INTERVAL_SECONDS 加上 POLL_JITTER_SECONDS 決定。收到 Ctrl+C 或 SIGTERM 時會在目前迴圈結束後停止。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\run_monitor_loop.py

若只想測監控，可單獨啟動本腳本；若要同時保留 Telegram 指令操作，請使用下一段的單一程序入口。 常駐排程遇到單輪未預期錯誤時會記錄錯誤並繼續下一輪，不會因單次例外直接退出。

啟動本機完整服務

本機完整服務會在同一個程序內啟動 Telegram 指令 long polling 與新房源監控排程， 共用同一個 SQLite repository。收到 Ctrl+C 或 SIGTERM 時會停止兩個 loop。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\run_local_service.py

查看監控狀態

這個指令只讀取 SQLite，不會抓 591，也不會發 Telegram；可在階段 5 的長時間驗收期間查看 active 訂閱、到期訂閱、待通知、成功與失敗通知數、總輪詢、有實際檢查訂閱的輪詢次數、 服務啟動次數、仍在 running 的服務紀錄，以及正常停止或失敗停止次數。 若資料庫 schema 過舊，工具會停止並提示先備份再初始化，不會自動 migration。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\monitor_status.py

驗證階段 5 長時間運行

階段 5 的正式關卡不是「服務可以啟動」，而是本機完整服務在驗收時間窗內累積至少 8 小時證據、 至少 90 次有實際檢查訂閱的監控輪詢、至少 2 次服務啟動紀錄、沒有未處理例外、沒有重複成功通知、 SQLite 完整，且人工抽查 20 筆 Telegram 圖片通知與房源一致。

開始前先做預檢。它只讀取 SQLite 與非秘密輪詢設定，不會抓 591，也不會發 Telegram。 預檢不會自動升級資料庫 schema；若顯示 schema 過舊，先依備份流程建立備份，再執行初始化資料庫。 若出現 warning，代表目前設定在最保守情況下可能需要跑超過 8 小時才能累積 90 次有效檢查。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\check_phase5_readiness.py

開始正式驗收前，先建立驗收時間窗檔案；後續驗證會只統計這個時間之後的監控輪詢。 工具會拒絕覆寫既有檔案，避免誤把正式驗收起始時間蓋掉。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\create_phase5_window.py --output .\outputs\phase5-window.json

驗收期間需至少做一次冷啟動驗證：啟動本機完整服務，確認監控開始後用 Ctrl+C 正常停止，再用同一個指令重新啟動並繼續跑到時間與檢查次數達標。 服務每次啟動與停止會寫入 service_runs；驗證腳本預設要求至少 2 次服務啟動， 且不能留下仍標記為 running 或 failed 的服務紀錄。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\run_local_service.py

跑完本機長時間服務後，先匯出圖片通知抽查表。工具會從已成功送出的圖片通知中挑出候選資料， 並拒絕覆寫既有檔案；若 schema 過舊，同樣會停止並要求先備份再初始化。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\export_phase5_image_audit.py --output .\outputs\phase5-image-audit.csv

開啟 CSV 後逐筆確認 Telegram 圖片是否對應同一個房源，並在 image_matches 欄填入 yes 或 no。 再用完成的抽查檔驗證階段 5；未達標時腳本會回傳 PHASE5_RUNTIME_VALIDATION_INCOMPLETE 並列出缺少或失敗的證據。 驗證腳本只讀取既有證據；若 schema 過舊，會回傳 PHASE5_RUNTIME_VALIDATION_NOT_READY，不會自動修改資料庫。 

$env:UV_CACHE_DIR = "$PWD\.uv-cache"
uv run --frozen python .\scripts\validate_phase5_runtime.py --window-file .\outputs\phase5-window.json --image-audit-file .\outputs\phase5-image-audit.csv

只有輸出 PHASE5_RUNTIME_VALIDATION_OK，才可將階段 5 標記為通過並進入 Railway 部署階段。

階段 1–4 驗收

  • 全新環境可依本文件安裝。
  • 缺少必要設定時，啟動檢查回傳失敗。
  • Ruff、pytest 與 compileall 全部通過。
  • Git 不包含 .env、Token、資料庫、日誌、快取或虛擬環境。
  • 重複抓取不會新增重複關係，通知失敗不會誤標成功。
  • 程序重啟、暫停恢復及備份重開後，訂閱與去重狀態保持一致。
  • Bot 指令、授權、確認、取消與通知模板已由自動測試覆蓋。
  • 私人 Telegram Bot 實測已通過，階段 4 關卡完成。
