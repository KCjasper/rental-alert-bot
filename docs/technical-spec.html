Technical Specification · v0.1

租屋情報機器人技術規格

本文件定義個人版 MVP 的架構、資料、介面、錯誤處理與驗收方式， 供下一階段直接進入開發。

1. 系統架構

Telegram 使用者
Bot 指令處理器
排程與訂閱服務
591 公開搜尋頁
SQLite 狀態資料庫
  • 執行環境：Python 3.12。
  • Bot：使用 Telegram Bot API；開發階段採 long polling，Railway 亦可先沿用。
  • 抓取：以 HTTP 請求讀取公開搜尋頁，再用 HTML 解析器抽取房源與公開首圖網址。
  • 儲存：SQLite 單檔資料庫，部署時掛載 Railway 持久磁碟。
  • 排程：單一常駐程序管理 Bot 與週期性搜尋工作。

2. 存取控制

所有 Telegram 更新必須先比對 AUTHORIZED_TELEGRAM_USER_ID。 不符合時不執行指令、不回傳訂閱資料，也不把內容寫入資料庫。

未授權使用者僅收到固定拒絕訊息；日誌只記錄必要的數字 ID 與時間， 不記錄 Token、完整訊息內容或個人資料。

3. Telegram 介面

輸入 行為
/start確認授權、顯示用途與貼入 591 搜尋網址的操作方式。
591 搜尋網址驗證網址、正規化查詢參數、讀取現有房源並建立待確認訂閱。
/subscriptions列出訂閱編號、名稱、狀態、最近檢查時間與最近結果。
/pause先提示使用者輸入訂閱編號；收到編號後暫停指定訂閱。也相容 /pause <id>。
/resume先提示使用者輸入訂閱編號；收到編號後恢復指定訂閱。也相容 /resume <id>。
/delete先提示使用者輸入訂閱編號，再要求二次確認後刪除訂閱；保留通知稽核紀錄。也相容 /delete <id>。
/test先提示使用者輸入訂閱編號；收到編號後立即抓取一次並回報解析數量，不改變已通知狀態。也相容 /test <id>。
/cancel取消首次全量通知、等待輸入編號或目前等待確認的動作。
/help顯示指令與常見錯誤說明。
確認確認目前等待中的首次全量通知或刪除動作；也接受 confirm、yes。

首次全量通知

  1. 網址驗證成功後先抓取一次，顯示目前房源總數。
  2. 使用者明確確認後才開始傳送，不將確認前的資料視為已通知。
  3. 按固定小批次傳送，每批之間節流，避免 Telegram 限流。
  4. 使用者傳送 /cancel 時取消等待確認或等待輸入編號的動作；首次加入尚未啟用時，訂閱會標成刪除。
  5. 若傳送中途被取消或失敗，已送出的房源仍記為已通知，未送出的房源保留待後續處理。
  6. 全量完成後，訂閱狀態轉為啟用並進入週期監看。

4. 591 網址與抓取規則

  • 僅接受 HTTPS 且主機名為 rent.591.com.tw 的租屋搜尋網址。
  • 拒絕帳號、刊登、管理、短網址及非搜尋結果頁。
  • 網址必須包含大於 0 的數字 region；拒絕帳密、非 443 連接埠及偽造子網域。
  • 正規化時移除追蹤、分頁及既有排序參數，固定參數順序，保留搜尋條件並加入 sort=posttime。
  • 公開搜尋頁以 div.item[data-id] 為房源節點，房源 ID 必須存在且同頁不得重複。
  • 標題及連結取自 .item-info-title a.link，租金取自 .item-info-price strong，其餘欄位取自 .item-info-txt。
  • 圖片只解析列表或詳情頁公開可見的首張房源圖；不登入、不解鎖相簿、不批量下載所有照片。
  • 圖片網址需正規化為 HTTPS，且只允許 591 或其公開圖片 CDN 網域；無法判定來源時不附圖。
  • 單頁必要結構缺失超過 20% 時，整頁判定解析失敗，不輸出部分結果供後續通知。
  • HTTP 逾時預設 20 秒；網路錯誤與 5xx 最多嘗試 3 次，等待 1、2 秒後重試。
  • 每次檢查預設間隔 300 秒，另加 0 至 30 秒隨機延遲。
不登入、不提交表單、不解 CAPTCHA、不使用代理輪替規避限制。 收到 403、429、人機驗證頁或明顯封鎖訊號時，立即停止本輪工作且不自動重試； 由後續排程延長檢查間隔並觸發告警。

5. 資料模型

資料表 主要欄位 用途
subscriptions id、name、source_url、normalized_url、status、檢查時間及軟刪除時間 保存搜尋訂閱及排程狀態；未刪除訂閱的正規化網址不可重複。
listings listing_id、url、image_url、title、price_monthly、位置、格局、坪數及首次／最近發現時間 保存已解析的最新房源摘要。
subscription_listings subscription_id、listing_id、first_seen_at、notified_at 建立訂閱與房源的去重關係。
notification_events id、訂閱與房源 ID、status、attempt_count、sent_at、錯誤代碼及訊息 追蹤發送成功、重試與失敗。
pending_actions action_type、payload、created_at、expires_at 保存首次全量及刪除動作的二次確認狀態。
monitor_runs started_at、completed_at、檢查數、成功數、失敗數、發送數與錯誤資訊 保存每次監控輪詢紀錄，用於階段 5 長時間驗收。

資料庫寫入需使用交易：只有 Telegram 回覆成功後才設定 notified_at，避免發送失敗卻永久漏掉通知。

  • 目前 schema 版本為 v4，透過 SQLite user_version 依序執行 migration。
  • 每次連線啟用外鍵、5 秒 busy timeout 與 WAL 模式，降低常駐服務讀寫互相阻塞。
  • 訂閱刪除採軟刪除，暫停、恢復及刪除都不會清空房源與通知歷史。
  • 已刪除訂閱不得出現在 /subscriptions，也不得再被 /pause、/resume、/delete 或 /test 操作。
  • 訂閱編號不可重用；刪除後重新建立相同搜尋條件時，使用新的下一個編號，避免舊通知與稽核紀錄混淆。
  • 資料庫層以 (subscription_id, listing_id) 主鍵去重；通知失敗只寫稽核事件，不寫 notified_at。
  • 圖片只保存公開圖片 URL，不保存圖片檔；若圖片 URL 更新，保留最新可用 URL。
  • 備份使用 SQLite online backup API，目標檔已存在時拒絕覆蓋，完成後執行完整性檢查。

6. 排程與去重流程

  1. 取得所有狀態為啟用且已到檢查時間的訂閱。
  2. 依正規化網址抓取搜尋結果，解析成一致的房源資料結構。
  3. 將房源與 subscription_listings 比對，找出首次出現且未通知的項目。
  4. 按照刊登時間由舊到新發送，使 Telegram 對話中的最新物件位於最下方。
  5. 每筆成功後立即提交通知紀錄；失敗項目留待帶上限的重試。
  6. 更新最近成功時間、結果數量及下一次檢查時間。

目前已提供一次性監控執行路徑 scripts/run_monitor_once.py； 常駐監控執行路徑 scripts/run_monitor_loop.py 已支援 5 分鐘輪詢、隨機延遲計算及 SIGINT/SIGTERM 優雅停止。 本機完整服務入口 scripts/run_local_service.py 可在同一程序中啟動 Bot 指令 long polling 與監控排程；圖片通知已支援公開首圖 URL、Telegram sendPhoto 與失敗退回文字。每次輪詢會寫入 monitor_runs， 供 scripts/validate_phase5_runtime.py 驗證 8 小時、90 次排程、 無未處理例外、無重複通知與資料庫完整性；實際 8 小時穩定性驗收仍在階段 5 後續工作中。

7. 錯誤處理與告警

情況 處理
網路逾時或 5xx最多嘗試 3 次並採指數退避；不改變既有去重狀態。
403、429 或 CAPTCHA不重試，停止本輪抓取；後續延長下次檢查時間並通知管理者。
解析結果突然為零若前次有資料，視為疑似改版，不將全部房源標記為消失。
必要欄位大量缺失判定解析器異常，保留原始狀態並發送告警。
圖片解析或傳送失敗不阻擋文字通知；記錄圖片失敗事件，房源仍可標記為已文字通知。
Telegram 限流讀取 Bot API 回傳的 retry_after 秒數後重試，最多 3 次。
SQLite 寫入失敗回滾交易、停止該輪通知並發送高優先告警。

同一問題連續失敗達 3 次時發送一次告警；恢復成功時再發送一次恢復通知，避免每 5 分鐘重複洗版。

  • Railway 部署時將 DATABASE_PATH 改為持久磁碟掛載路徑。

9. 驗收測試

案例 通過條件
合法搜尋網址成功正規化、抓取並顯示現有房源數。
惡意或錯誤網址拒絕非 591、非 HTTPS、非租屋搜尋頁及偽造子網域。
首次全量確認確認前不發送；確認後分批送完；取消後停止剩餘項目。
新房源去重同一房源在後續輪詢與程序重啟後都不重複通知。
房源圖片可附帶公開首圖；圖片與房源 ID 人工抽查一致，圖片失敗時仍有文字通知。
暫停與恢復暫停期間不抓取；恢復後保留既有去重資料。
未授權帳號不能建立、查看或操作任何訂閱。
Telegram 發送失敗房源保持未通知狀態，稍後可重試。
591 改版或封鎖不誤發、不清空狀態，並在門檻後發送告警。
服務重啟SQLite 資料完整，排程與待通知房源可繼續處理。

10. 部署前置條件

  • 由使用者透過 BotFather 建立私人 Telegram Bot，Token 不經聊天傳遞。
  • 確認個人 Telegram User ID，填入 Railway 的秘密變數。
  • 建立 Railway Hobby 專案、常駐服務及持久磁碟。
  • 設定程序健康檢查、重啟策略及支出上限。
  • 部署前重新確認 Railway 當期價格與 591 當期服務規範。
