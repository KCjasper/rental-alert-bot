Development Process · Definition of Done

產品開發流程表

後續開發一律依照本流程逐階段進行。每個階段必須留下可檢查的成果， 並通過階段關卡後才能進入下一階段；不能只因為「程式可以啟動」就視為產品完成。

流程版本：v1.0

執行原則

一次只完成一個階段

每階段先實作、測試、記錄證據，再更新進度。未通過關卡不得宣告完成。

測試結果必須可重現

不能只靠口頭確認；測試指令、案例與結果需保留在專案中。

真實資料最後驗證

單元測試通過後，仍須用真實 591 搜尋網址與私人 Telegram Bot 驗證。

第三方限制優先遵守

遇到封鎖、限流或 CAPTCHA 必須停止並告警，不以規避方式強行運作。

流程總覽

1. 開發準備 2. 591 資料層 3. 儲存與去重 4. Telegram Bot 5. 排程與整合 6. 雲端部署 7. 穩定性驗收 完成標準

階段流程與關卡

1

開發準備與專案骨架

建立可重現、可測試、可安全保存設定的開發環境。

工作內容

  • 建立 Python 3.12 專案與依賴管理。
  • 建立原始碼、測試、資料與日誌目錄。
  • 加入設定讀取、結構化日誌與啟動檢查。
  • 準備 CI 測試流程與本機啟動指令。

必要產物

  • 套件與版本鎖定檔。
  • 應用程式進入點與設定模組。
  • 測試框架與至少一個冒煙測試。
  • 開發操作文件。

驗證證據

  • 全新環境可依文件完成安裝。
  • 缺少 Token 時明確拒絕啟動。
  • 測試與靜態檢查指令皆成功。
  • Git 無敏感設定或執行資料。
階段關卡：另一個乾淨環境可在 15 分鐘內安裝並執行測試，且不需修改程式碼。
2

591 網址驗證與資料解析

先證明能穩定辨識合法搜尋頁與房源資料，再串接 Bot。

工作內容

  • 驗證並正規化 591 搜尋網址。
  • 實作 HTTP 用戶端、逾時與退避。
  • 解析房源 ID、標題、租金、位置、格局、坪數、時間與連結。
  • 辨識封鎖頁、CAPTCHA 與頁面結構異常。

必要產物

  • 網址驗證器。
  • 591 抓取器與解析器。
  • 去識別化 HTML 測試樣本。
  • 解析結果資料型別。

驗證證據

  • 合法網址可解析，偽造網域會被拒絕。
  • 固定樣本測試涵蓋正常、空結果與改版情境。
  • 真實搜尋頁抽查至少 3 組不同條件。
  • 解析欄位與網頁顯示一致。
階段關卡：3 組真實搜尋條件皆能取得房源，抽查 20 筆的房源 ID、租金與連結正確率為 100%。
3

SQLite 儲存、狀態與去重

確保服務重啟、失敗重試與多組訂閱都不會造成重複通知或資料遺失。

工作內容

  • 建立資料表、索引與初始化流程。
  • 實作訂閱新增、暫停、恢復與刪除。
  • 實作房源首次發現與通知狀態交易。
  • 保存待確認動作與發送稽核紀錄。

必要產物

  • 資料庫 schema 與 migration。
  • Repository／資料存取層。
  • 測試用臨時資料庫。
  • 備份與還原說明。

驗證證據

  • 同一房源重複輸入只建立一筆關係。
  • 發送失敗時不寫入已通知時間。
  • 程序重啟後訂閱與去重狀態完整。
  • 暫停／恢復不會清除歷史。
階段關卡：自動測試模擬重複抓取與程序重啟，所有房源都恰好通知一次。
4

Telegram Bot 操作與通知

完成私人帳號授權、訂閱管理與清楚可讀的房源通知。

工作內容

  • 實作單一 Telegram User ID 授權。
  • 完成全部規格指令與錯誤提示。
  • 實作貼入網址、首次全量確認與取消。
  • 處理 Telegram 限流與發送重試。

必要產物

  • Bot 指令處理器。
  • 通知訊息模板。
  • 首次全量批次服務。
  • 授權與訊息單元測試。

驗證證據

  • 非授權帳號無法讀寫任何資料。
  • 所有指令皆有成功與錯誤案例。
  • 取消全量通知後不再繼續發送。
  • 通知欄位完整且連結可開啟。
階段關卡：使用私人測試 Bot 完整走過新增、確認、取消、暫停、恢復、測試與刪除流程，結果全部符合規格。
5

排程、整合與本機端到端測試

把抓取、資料庫與 Bot 組成可長時間運作的單一服務。

工作內容

  • 實作 5 分鐘輪詢與 0–30 秒隨機延遲。
  • 整合新房源偵測與通知交易。
  • 加入優雅停止、重啟恢復與健康狀態。
  • 加入連續失敗告警與恢復通知。

必要產物

  • 排程器與應用服務。
  • 端到端測試腳本。
  • 健康檢查或狀態輸出。
  • 故障注入測試案例。

驗證證據

  • 模擬新房源後能在下一輪通知。
  • 相同結果連續檢查不重複通知。
  • 網路、Telegram、SQLite 失敗皆可恢復。
  • 連續 8 小時本機運行無崩潰。
階段關卡：本機連續 8 小時完成至少 90 次排程，無未處理例外、無重複通知、無資料庫損壞。
6

Railway 部署與營運設定

把通過本機驗收的版本部署成可自動重啟、可保存資料的雲端服務。

工作內容

  • 建立 Railway 服務與秘密變數。
  • 掛載 SQLite 持久磁碟。
  • 設定啟動命令、健康檢查與重啟政策。
  • 設定支出上限並驗證日誌不洩漏秘密。

必要產物

  • 部署設定與操作文件。
  • Railway Variables 清單。
  • 資料備份步驟。
  • 回退至上一版本的方法。

驗證證據

  • 重新部署後 SQLite 資料仍存在。
  • 程序異常結束後能自動恢復。
  • Bot 可從雲端持續接收與發送訊息。
  • 日誌與 Git 都沒有 Token。
階段關卡：執行一次重新部署與一次強制重啟後，訂閱、去重資料及 Bot 功能皆保持正常。
7

真實環境穩定性驗收

用實際搜尋條件觀察完整運行週期，達標後才能宣告產品完成。

工作內容

  • 建立至少 2 組真實搜尋訂閱。
  • 連續觀察通知、去重、告警與恢復。
  • 執行最終安全、備份與回退演練。
  • 記錄已知限制與營運處理方式。

必要產物

  • 72 小時運行紀錄。
  • 最終驗收報告。
  • 已知限制清單。
  • 日常維護與故障處理手冊。

驗證證據

  • 服務可用率至少 99%。
  • 95% 排程在 10 分鐘內完成新房源檢查。
  • 零重複通知、零漏存成功通知。
  • 告警、重啟、備份與還原演練成功。
階段關卡：72 小時真實運行達成全部量化標準，且沒有尚未處理的重大缺陷。

產品完成標準

只有下列所有條件都通過，產品狀態才能從「開發中」改為「完成」。 任一必要條件未通過，都只能標示為測試版或待修正。

功能完成

  • 可新增多組合法的 591 租屋搜尋網址，錯誤網址會被明確拒絕。
  • 可完成首次房源數量確認、分批全量通知與中途取消。
  • /start、/subscriptions、/pause、/resume、/delete、/test、/cancel、/help 全部可用。
  • 新房源通知包含標題、租金、地區、格局、坪數、刊登時間與有效連結。

正確性與可靠性

  • 20 筆人工抽查房源的 ID、租金與連結正確率為 100%。
  • 相同房源在重複輪詢、重新部署及程序重啟後都不會重複通知。
  • Telegram 發送失敗時不會誤標為已通知，恢復後可重新傳送。
  • SQLite 資料可備份、還原，且重新部署後不遺失。

安全與合規

  • 只有指定的 Telegram User ID 能操作 Bot 或查看訂閱。
  • Git、日誌、錯誤訊息與測試資料中均沒有真實 Token。
  • 系統不登入 591、不解 CAPTCHA、不規避限流或封鎖。
  • 403、429、CAPTCHA、解析異常與資料庫錯誤都會安全停止並告警。

上線品質

  • Railway 健康檢查、自動重啟、持久磁碟與支出上限皆已設定。
  • 連續 72 小時真實運行，可用率至少 99%。
  • 95% 的檢查週期在 10 分鐘內完成；延遲或失敗會留下可追查紀錄。
  • 沒有 P0／P1 重大缺陷；P2 缺陷必須有明確處理方案且不影響核心用途。

文件與交付

  • 安裝、測試、部署、備份、還原與回退步驟都已驗證。
  • 自動測試全部通過，且包含網址安全、去重、權限與失敗重試案例。
  • 已知限制與第三方服務風險有清楚記錄。
  • 最終驗收報告附上測試結果與 72 小時運行證據。

缺陷分級與發布規則

等級 定義 發布規則
P0 阻斷 洩漏 Token、未授權者可操作、資料庫毀損、大量錯誤通知。 立即停止服務，修復並重新完整驗收。
P1 重大 無法抓取、無法通知、持續重複通知、重啟後狀態遺失。 不得發布或宣告完成。
P2 一般 部分非必要欄位缺失、個別錯誤訊息不清楚、次要操作不便。 可在不影響核心用途時附處理計畫發布。
P3 改善 版面、文案、日誌可讀性或非必要效能改善。 可列入後續版本。

每次開發工作的完成定義

程式層

實作符合規格、錯誤情境已處理、沒有秘密或除錯殘留。

測試層

新增或更新相關測試，完整測試與靜態檢查皆通過。

文件層

使用方式、設定或行為有變更時，同步更新 HTML 文件。

版本層

Git 差異只包含本次工作，提交訊息清楚，工作樹保持乾淨。

開發過程若發現 591 實際頁面行為與現有規格不同，必須先更新規格與驗收條件， 再繼續實作，不能默默改變產品行為。
